TCP

TCP被称为是面向连接的，这是因为在一个应用进程可以开始向另一个应用进程发送数据之前，这两个进程必须先相互“握手”，即它们必须相互发送某些预备报文段，以建立确保数据传输的参数。（TCP是可靠的，建立连接保证可靠性，建立连接的过程叫做握手）

三次握手

建立连接前服务端处于监听状态，客户端准备建立连接，先发送一个同步包，此时客户端的状态变为同步发送，由于TCP是双工传输（TCP 在传输上是双工传输，不区分 Client 端与 Server 端）服务端会回复一个同步包以及确认包，此时服务端的状态变成同步已接收，客户端收到确认包之后状态变为已建立连接，并向服务端发送确认包，服务端收到确认包后状态也变为以建立连接。此时建立连接完成，双发随时可以进行数据传输。（三次握手是为了建立双方的连接，需要记住客户端和服务端连接状态的变化）

为什么是三次握手

如果是两次

  1. 服务端只知道客户端有发送的能力，不知道是否有接收的能力
  2. 客户端发出请求，由于网络堵塞，长时间未能收到服务端的应答，客户端认为报文丢失便重新发送，这一次连接成功。过段时间之后，服务端会收到堵塞的报文，就会再次进行应答，如果此时客户端进行连接但是并不会发送数据造成资源浪费。

过多的握手是多余的，第二次握手已经答应建立连接。

SYN Flood洪水攻击

服务端在第二次握手之后无法收到客户端的报文（第三次握手无法完成），这种情况下服务端一般会重试并等待一段时间后丢弃这个连接，这个时间一般为30s-2min。如果有大量等待丢失的情况发生将会造成巨大的资源消耗。

针对少量的洪水攻击可以通过设置减少丢弃时间，或者给每个ip地址分配一个cookie，如果发现短时间内收到同一ip地址的多个重复报文，认为受到攻击，丢弃改ip地址的包。

四次挥手

由于TCP连接是全双工的，因此每个方向都必须单独进行关闭。这个原则是当一方完成它的数据发送任务后就能发送一个FIN(finish)来终止这个方向的连接。收到一个 FIN只意味着这一方向上没有数据流动，一个TCP连接在收到一个FIN后仍能发送数据。首先进行关闭的一方将执行主动关闭，而另一方执行被动关闭。

客户端发送FIN包，关闭客户端到服务端的数据传送，服务端收到后，回复给客户端一个确认包。当服务端数据发送完毕，服务端向客户端发送结束包，表示服务端没有数据要传送了，客户端收到后回复给服务端确认包，进入TIME_WAIT状态，要等待 2 倍的最大报文段生存时间，来保证链接的可靠关闭，之后才会进入 CLOSED 关闭状态。而服务端收到 ACK 后直接就进入 CLOSED 状态。

为什么需要等待 2 倍最大报文段生存时间之后再关闭链接
  1. 保证 TCP 协议的全双工连接能够可靠关闭
  2. 保证这次连接的重复数据段从网络中消失，防止端口被重用时可能产生数据混淆
